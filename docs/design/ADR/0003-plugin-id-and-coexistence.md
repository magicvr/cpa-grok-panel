# ADR 0003：插件 ID 与旧插件共存

- 状态：已接受
- 日期：2026-07-15

## 背景

部署环境可能已有 id 为 `grok-panel` 的旧插件。新项目需要独立安装、灰度和回滚，不能覆盖旧插件文件或状态。

## 决策

新插件 id 使用 `cpa-grok-panel`。所有可命名资源均从该 id 派生：

- 管理 API 前缀：`/plugins/cpa-grok-panel/api/v1`
- 面板入口：`/plugins/cpa-grok-panel/`
- 静态资源命名空间：`cpa-grok-panel`
- state 子目录：由 CPA 分配的该插件专属 data dir
- 日志/指标前缀：`cpa_grok_panel`

不得声明旧插件 id、复用旧插件状态目录或注册相同管理路由。

## 共存规则

- 允许新旧插件同时安装和读取账号状态。
- 灰度期间只能指定一个插件为 auth 写操作责任方。
- 当 `write_mode=read_only` 时，本插件禁止降权、启停、恢复和删除，只提供观察能力。
- 切换责任方前应停止旧插件写操作、刷新 CPA auth 清单，再启用本插件 `write_mode=managed`。
- 不导入旧插件内部 state；usage 计数从本插件启用时开始，界面明确显示统计起点。
- 回滚时先将本插件切回只读并排空操作，再恢复旧插件写责任。

## 冲突检测

若 CPA 能暴露已注册插件能力，本插件应检测同类写插件并提示风险；若宿主不能检测，则安装/运维文档要求管理员显式确认。插件不能仅凭旧 id 的存在假定冲突，也不能强制卸载其他插件。

## 后果

新旧插件可以独立发布和回滚，但灰度期间需要人为确定写责任，且两边 usage 统计不会天然连续。

## 被拒绝方案

- 沿用 `grok-panel` id 原地升级：无法安全并存或快速回滚。
- 自动迁移旧插件状态：违反 clean-room 边界，并可能错误解释私有 schema。
- 双写 auth 设置：存在竞态和覆盖风险。
