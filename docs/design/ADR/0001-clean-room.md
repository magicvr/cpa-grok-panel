# ADR 0001：采用 Clean-room 独立设计

- 状态：已接受
- 日期：2026-07-15

## 背景

目标产品在功能类别上可能与既有 `grok-panel`、`cpa-plugin-grok-panel` 或其他实现相似，但本仓库要求形成独立、可审计的设计来源。

## 决策

本项目采用 clean-room 方法：只依据公开产品需求、CPA 经授权的正式契约以及本仓库自行产生的设计实现。禁止读取、复制、粘贴、反编译、改写或通过行为细节复刻任何既有相关插件源码。

当前阶段只输出设计文档、接口契约、目录规划和非可编译流程描述，不输出业务实现。

## 约束措施

- 设计记录注明需求来源与待验证假设。
- CPA ABI 未经官方资料或实验确认的字段标记为“目标契约”。
- 代码评审检查提交来源，不接受来源不明的大段实现。
- 不把旧插件 state、路由或私有协议作为兼容目标。
- 迁移只通过人类确认的公开数据契约进行，不读取旧插件内部文件格式。

## 后果

优点：知识产权边界清晰；架构可按当前需求重新约束；新旧插件能独立演进。

代价：无法继承旧实现的边缘行为；CPA 集成细节必须重新验证；初期可能需要更多契约测试。

## 被拒绝方案

- 从旧插件 fork 后重构：违反 clean-room 约束。
- 兼容旧插件未公开 state 格式：会形成实现耦合和来源风险。
- 复刻旧 UI 行为作为验收标准：不必要且无法证明独立设计。
