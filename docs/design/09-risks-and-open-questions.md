# 09. 风险与开放问题

## 1. 主要风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| CPA ABI 与目标契约不一致 | 功能无法落地 | capability 协商；缺则降级；实现前实测 |
| usage 无稳定 event_id | 重复累计 | 阻断项；与 CPA 确认；否则弱一致性并明示 |
| usage 无可信 auth 身份 | 无法降权 | 不猜测；指标暴露未归因事件 |
| 失败归责不准 | 误降权 | 保守 allowlist；人工可恢复 baseline |
| 与旧 grok-panel 双写 | 优先级互相覆盖 | 单一写责任；write_mode |
| host 无条件写 | 覆盖竞态 | 降级或禁用危险写 |
| 直接操作 auth 文件诱惑 | 安全/兼容灾难 | 架构禁止；评审拦截 |
| token 上限被误解为账单 | 错误运营决策 | UI 文案强调“运维分母” |
| 检查产生真实费用 | 成本 | 严格限流、最小 profile、确认文案 |
| state 损坏 | 丢失统计 | bak + 诊断模式；auth 真相仍在 CPA |

## 2. 开放问题（供讨论，≥8）

1. **CPA 是否提供稳定 `auth_file_id`？** 若只有文件名，重命名如何处理？
2. **`usage` 事件是否有全局唯一 `event_id` 与 at-least-once 重试？**
3. **失败归责字段是否存在？** 没有时 allowlist 如何定义？
4. **`priority=-100` 是否为 CPA 正式低优先级语义？** 合法范围？
5. **host 是否支持条件写（expected_revision）？** 没有能否接受 last-write-wins？
6. **人工检查如何锁定指定 auth 而不被调度替换？** `host.auth.invoke` 是否存在？
7. **删除 API 是否要求精确文件名 + 管理鉴权？** 能力缺失时 MVP 是否砍掉清理？
8. **管理角色模型：** 仅单一管理密钥，还是 viewer/operator/admin？
9. **灰度期间写责任切换流程** 是否接受人工 runbook，还是要插件互斥检测？
10. **usage 统计起点：** 是否接受“新插件从启用日起算”，不做历史迁移？
11. **Responses 实测是否计入失败降权？** 完全交给 CPA usage 归责是否足够？
12. **首发是否只做 Linux amd64 + 只读 MVP，再开写？**
13. **前端技术选型：** 轻量原生 / 小型框架？是否嵌入静态资源？
14. **state 是否允许短 debounce 丢数窗口？** 可接受秒级？
15. **与旧插件 UI 并存时菜单命名** 如何避免运维点错？

## 3. 建议的决策顺序

1. 先钉死 CPA 身份与 usage 契约（问题 1–3）
2. 再钉死写能力（4–7）
3. 再定 MVP 砍砍法（12–13）
4. 最后定灰度 runbook（9–11）

## 4. 明确不做的“捷径”

- 不读旧插件源码“对齐行为”
- 不解析 auth JSON 秘密字段做套餐猜测
- 不在浏览器存管理密钥
- 不自动巡检、不自动删除
